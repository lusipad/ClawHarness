## 验收目标

V2 的验收目标不是证明“新增了哪些功能”，而是证明系统已经具备以下闭环能力：

- 能处理真实复杂工程任务
- 能以多 agent 方式执行并保持可观察
- 能在反馈、失败和重启后恢复
- 能通过聊天或 bot-view 查看并介入运行
- 能扩展到多 provider、多渠道和图片场景
- 能在复制部署时保持稳定、低维护成本

## 验收原则

- 每条验收标准都必须由可观察证据证明
- 复杂任务只有在父子 run 关系、工件链和状态迁移可追踪时才算通过
- 多 agent 只有在 agent 角色、结果、失败和汇聚过程可见时才算通过
- 聊天与 bot-view 只有在“可查看”和“可审计”同时成立时才算通过
- 多 provider 支持只有在核心状态机保持统一时才算通过
- 图片支持只有在图片真正进入任务上下文并产生可追踪影响时才算通过
- Docker / 原生 / 模板化部署属于工程化验收的一部分，不能后置到无限期

## 验收标准

### AC-01：复杂任务父子 RunGraph

阶段：
- V2.0 Core

要求：
- 系统必须能把一个复杂任务映射为一个父 run 与多个子 run，而不是把全部复杂性压在一个线性 run 里

给定：
- 一个明确需要拆分的复杂任务

当：
- 任务被分析、规划并进入执行

则：
- 只创建一个父 run
- 至少创建两个有明确职责的子 run
- 子 run 之间的依赖关系可见
- 每个子 run 都能挂载自己的工件与状态
- 重放相同触发事件时不会重复创建父 run

证据：
- RunGraph 记录
- 父子 run 状态迁移链
- 去重审计记录

### AC-02：统一状态机与检查点恢复

阶段：
- V2.0 Core

要求：
- V2 必须有统一的阶段状态机和检查点恢复机制

给定：
- 一个进入 planning、coding、review、publish 任一阶段的任务

当：
- 服务进程重启或 worker 中断后恢复

则：
- 系统能从最近检查点恢复
- 不重复创建 branch、PR、评论回复等外部副作用
- 恢复动作进入统一审计链

证据：
- checkpoint 记录
- 恢复前后状态比对
- 无重复副作用的外部系统记录

### AC-03：多 agent 自动协作

阶段：
- V2.0 Core

要求：
- OpenClaw 必须能自动组织多个 agent 参与同一复杂任务

给定：
- 一个需要规划、实现、审查和验证的任务

当：
- 复杂任务进入执行态

则：
- 至少包含 planner、executor、reviewer、verifier 四类角色中的多种角色
- agent 结果会被系统汇总，而不是散落在日志中
- agent 失败时可记录重试、替换或升级人工决策

证据：
- agent 树或 agent 列表
- agent 级工件
- agent 失败与补救审计

### AC-04：复杂任务主闭环

阶段：
- V2.0 Core

要求：
- V2 必须完成真实复杂任务从任务源到 PR 的完整闭环

给定：
- 一个真实复杂任务，涉及多文件或多步骤修改

当：
- 从 Azure DevOps 或后续支持的任务源触发执行

则：
- 任务被拆解、执行、检查、汇总
- 分支与 PR 成功创建
- 结果状态进入 `awaiting_review`、`awaiting_ci` 或等价阶段
- 全过程可从父 run 追踪到子 run 和关键工件

证据：
- 真实 PR 链接
- 运行审计链
- 变更摘要与检查结果

### AC-05：PR feedback 同父 Run 恢复

阶段：
- V2.0 Core

要求：
- PR 评论恢复必须落在同一父 run 上下文，而不是新起一条无关链路

给定：
- 一个已经由系统创建的 PR
- 一条新的评审评论或 review 请求

当：
- PR feedback 被系统接收并处理

则：
- 系统能从 `pr_id` 找回同一父 run
- 会创建新的修复子 run，而不是新的父 run
- 修复完成后更新同一个 PR

证据：
- `pr_id -> parent_run_id` 映射
- 修复子 run 记录
- PR 更新时间线

### AC-06：聊天观察与介入

阶段：
- V2.1 Channels

要求：
- 用户必须能通过聊天渠道查看状态并进行受控介入

给定：
- 一个正在运行或阻塞中的任务
- 已接入的 Rocket.Chat 或 Weixin 渠道

当：
- 用户发送 `status`、`detail`、`pause`、`resume`、`add-context`、`escalate` 等命令

则：
- 系统返回当前状态、阶段、最近事件与关键工件摘要
- 用户命令被归一化并写入审计链
- 受控操作会影响正确的父 run 或子 run

证据：
- 聊天消息记录
- 命令归一化记录
- 运行审计与状态变化

### AC-07：GitHub 工作流闭环

阶段：
- V2.2 Providers

要求：
- GitHub 必须作为与 Azure DevOps 并列的一等工作流来源

给定：
- 一个 GitHub issue 或 PR 修复请求

当：
- 系统接收并执行该任务

则：
- 能完成 issue / task 到 branch 到 PR 的闭环
- 能正确处理 GitHub review comment 与 checks failure
- 核心 run graph、状态机、工件模型与 Azure DevOps 保持一致

证据：
- 真实 GitHub PR 链接
- GitHub 事件与 run 映射记录
- GitHub checks / review 恢复证据

### AC-08：CI failure 修复或升级

阶段：
- V2.0 Core

要求：
- CI 失败必须回到同一父 run，并进入“自动修复”或“升级人工”之一

给定：
- 一个与父 run 已绑定的构建或 checks 失败事件

当：
- 系统处理 CI failure

则：
- 能找回同一父 run
- 会创建 repair 子 run 或写入人工升级决策
- 若执行自动修复，则必须保留修复前后检查证据

证据：
- `ci_run_id` 或等价检查标识到父 run 的映射
- repair 子 run 记录
- 修复检查或升级审计

### AC-09：bot-view 运行面

阶段：
- V2.0 Core

要求：
- bot-view 必须成为标准运行观察面

给定：
- 一个正在运行、等待评审、等待 CI、失败或完成的任务

当：
- 操作者打开 bot-view

则：
- 能看到父 run、子 run、当前阶段、最近事件、关键工件、外部链接
- 至少能定位当前阻塞点、最近失败、最近 agent 结论
- 数据展示默认做敏感信息裁剪

证据：
- bot-view 页面或接口输出
- 工件与状态对照验证

### AC-10：可观测性与审计完整性

阶段：
- V2.0 Core

要求：
- V2 的所有关键动作都必须进入统一审计与可观测链路

给定：
- 一次完整运行，包含至少一次 agent 执行、一次状态迁移、一次检查和一次外部系统交互

当：
- 审计系统被查询

则：
- 可按 task、run、agent、PR、CI、channel 查询关键事件
- 审计链能解释“做了什么、什么时候做、为什么失败或阻塞”
- 日志、摘要、链接和关键工件之间可互相定位

证据：
- 事件时间线
- 审计查询结果
- 关键工件索引

### AC-11：图片输入与识别闭环

阶段：
- V2.1 Channels

要求：
- 系统必须能处理图片类任务输入

给定：
- 一张截图、缺陷图片或带图问题报告

当：
- 图片通过聊天、任务附件或 bot-view 上传进入系统

则：
- 图片被挂载为统一工件
- agent 能访问图片上下文并给出分析或修复建议
- 图片相关结论进入父 run 证据链

证据：
- 图片工件记录
- 图像分析输出
- 与后续修复任务的关联记录

### AC-12：Provider-neutral 核心工作流

阶段：
- V2.2 Providers

要求：
- 多 provider 支持不能通过复制两套工作流来实现

给定：
- Azure DevOps 与 GitHub 两种 provider 配置

当：
- 审查 flow、状态机和适配器实现

则：
- 核心编排逻辑不直接依赖 provider 私有动作名
- provider 差异收敛在适配器边界
- 共享阶段名、共享工件模型、共享恢复规则保持一致

证据：
- flow / runtime 静态审查结果
- adapter 契约
- 跨 provider 验证记录

### AC-13：Skill 自动选择与审计

阶段：
- V2.3 Skills

要求：
- skill 必须可自动选择、可解释、可审计、可回滚

给定：
- 一组具有不同任务类型适配条件的 skill 包

当：
- 系统为复杂任务选择 skill

则：
- 能记录选择了哪些 skill、版本号、来源和触发原因
- 失败时可回退到安全默认路径
- 人工可查看和审计 skill 选择依据

证据：
- skill registry 记录
- 选择日志
- 回退或禁用验证记录

### AC-14：安全与策略护栏

阶段：
- V2.0 Core

要求：
- 多 agent、多渠道和多 provider 场景下仍需保持最小安全边界

给定：
- 受保护分支、密钥配置、聊天指令和外部 provider 访问

当：
- 系统运行复杂任务或执行人工介入命令

则：
- 不允许绕过保护分支策略
- 不允许默认直接合并
- 敏感配置通过环境变量或等价安全方式注入
- 聊天与 bot-view 的高风险动作有显式审计与权限边界

证据：
- 策略配置
- 运行审计
- 脱敏验证记录

### AC-15：模板化部署与快速复制

阶段：
- V2.0 Core

要求：
- 新实例部署必须以稳定默认字段和模板渲染为主，而不是复制整块版本敏感配置

给定：
- 一个全新 Docker 或原生部署环境

当：
- 使用部署模板完成安装与启动

则：
- 只需填写少量稳定字段即可完成部署
- 系统能自动生成内部版本敏感配置
- 健康检查通过
- 文档能说明如何复制、升级和恢复默认配置

证据：
- 部署模板
- 健康检查输出
- 真实新实例部署记录

### AC-16：长期运行与资源回收

阶段：
- V2.3 Skills

要求：
- V2 必须能在长期运行中控制资源和历史工件膨胀

给定：
- 多个长期 run、图片工件、聊天事件和 skill 包历史

当：
- 系统执行归档、清理或版本切换

则：
- 不影响活跃 run 的恢复能力
- 可配置地清理旧工件和临时工作区
- 历史 run 与关键证据仍可被审计

证据：
- 清理策略
- 归档记录
- 清理前后一致性验证

## 真实环境 E2E 场景

### E2E-01：Azure DevOps 复杂任务闭环

步骤：

1. 从 Azure DevOps 触发一个真实复杂工作项
2. 系统生成父 run 和多个子 run
3. agent 协作完成修改、检查、提交和 PR
4. bot-view 或聊天可查看运行细节

通过标准：

- 真实 PR 被创建
- 父子 run 关系可见
- 至少一次 agent 汇聚结果可见

### E2E-02：PR feedback 恢复闭环

步骤：

1. 在系统创建的 PR 上添加新的评审意见
2. 系统恢复同一父 run 并创建修复子 run
3. 修复结果更新到同一 PR

通过标准：

- 不创建新的父 run
- 修复子 run 和工件可追踪
- PR 更新时间线完整

### E2E-03：CI failure 恢复闭环

步骤：

1. 让一次关联构建或 checks 失败
2. 系统读取失败摘要并尝试自动修复
3. 无法修复时升级人工

通过标准：

- 同一父 run 被正确找回
- 自动修复或升级人工二选一明确发生
- 结果有完整审计

### E2E-04：聊天查看与介入

步骤：

1. 在 Rocket.Chat 或 Weixin 中查询某个 run 状态
2. 追加新的上下文或约束
3. 执行暂停、恢复或升级人工

通过标准：

- 命令作用到正确 run
- 聊天回复包含状态摘要和关键链接
- 所有命令都有审计记录

### E2E-05：图片问题闭环

步骤：

1. 上传截图或图片问题报告
2. 系统把图片作为工件挂到父 run
3. agent 基于图片给出分析并进入修复流程

通过标准：

- 图片工件可追踪
- 图像分析结果可追踪
- 后续修复流程与图片证据相关联

### E2E-06：GitHub 闭环

步骤：

1. 从 GitHub issue 或 review request 触发
2. 系统完成修改并创建或更新 GitHub PR
3. GitHub checks 或 review comment 触发恢复

通过标准：

- GitHub 主链路跑通
- GitHub 恢复链路跑通
- 核心状态机与工件模型不分叉

### E2E-07：重启恢复

步骤：

1. 在长任务执行过程中重启 gateway 或 bridge
2. 系统从检查点恢复运行
3. 验证外部副作用未重复

通过标准：

- run 可恢复
- 无重复 branch / PR / 评论回复
- 恢复动作被审计

## 可观测性专项验收

V2 至少要满足以下可观测性要求：

- 任一活跃任务都能在 bot-view 或等价接口中看到当前阶段
- 任一 agent 的最近结果、失败原因和后续建议可查询
- 任一外部链接，例如 work item、PR、build、chat message，都能回溯到父 run
- 任一阻塞态都能给出结构化阻塞原因和建议动作
- 任一人工介入动作都有明确操作者、时间和影响范围记录

## 稳定性与恢复专项验收

V2 至少要满足以下稳定性要求：

- 重复 webhook 或重复命令不会创建重复父 run
- 长任务恢复后不会重复外部副作用
- 活跃工作区与历史工作区可区分管理
- 活跃 run、历史 run、工件和临时资源有清晰回收策略

## 证据要求

每个 AC 与 E2E 场景都至少需要一种证据，推荐组合如下：

- 真实外部系统链接，例如 PR、issue、work item、build、chat message
- 运行时数据库记录
- bot-view 截图或接口输出
- 工件文件，例如计划、检查结果、图片分析结果
- 审计事件链
- 部署与健康检查记录

## 当前结论

截至 `2026-04-06`，V2 的 Azure 主链路验收已经跨过了“只证明 happy path”的阶段，真实证据已经覆盖复杂任务主闭环、父子 run 图、PR feedback 恢复、CI failure 恢复、只读运行面和本地部署。当前剩余工作主要集中在 GitHub live webhook 联调以及更广泛的渠道 / 部署覆盖，而不是核心运行时能力缺失。
